# Section: Core experience & accomplishments (mineable facts)

## Canonical heading

`## Core experience & accomplishments (mineable facts)`

## What this section captures

Per-role facts the drafting agent will pull from to populate resume bullets. Each role gets: scope (budget, headcount, structure), what the user **personally** built or shipped (with mechanism, not just outcomes), quantified outcomes, named systems, stakeholder altitude.

**This is the longest section.** It's the source of every resume bullet. Get it right and the rest of the workflow gets sharper.

## Questions to cover (per role, working back through their last ~10 years)

For each role, ask in this order:

1. **Role identity.** Company, HR title, *functional* title (what you actually were), dates (month/year — month matters for tenure stories), location, remote/hybrid/onsite.
2. **Scope at the role.** Budget owned ($M), team size and structure (direct reports / dotted lines / total org under you), stack scope (cloud + tooling), regulatory boundary if any (FedRAMP / HIPAA / etc.).
3. **What you actually built or shipped.** Get **3–7 concrete things** per role. For each, push for:
   - **Mechanism:** how was it done? ("led a team" doesn't count — what did the team build, what did the user personally contribute?)
   - **Named system:** does it have a name people can search for?
   - **Why it mattered:** what problem was it solving?
4. **Quantified outcomes.** For each shipped thing, the numeric impact. Push for: how much, how many, over what timeframe, vs what baseline. If they don't remember exactly, mark `TBD` — never guess.
5. **Stakeholder altitude.** Who did you brief? CIO/CTO/VP/Director/peer engineers? Decision authority you held?
6. **Outcomes at the role (overall).** Top-line numbers attributable to the user. Promotions, expansion of scope, recognition.
7. **Edge cases & anti-patterns.** Anything ambiguous about scope or title? Anything past tailorings got wrong about this role that we should never repeat? (E.g., a fabricated team structure that needs an explicit anti-pattern note.)
8. **Why you left.** Optional but useful — captures pattern over time.

After the current/most-recent role, ask whether to deep-dive earlier roles or compress them. For earlier roles (3+ jobs back), one-paragraph compressed entries are usually fine unless something specific (e.g., a from-scratch build) deserves to be surfaced.

## Output structure (draft format)

```
## Core experience & accomplishments (mineable facts)

### Current role — <Company>, <Functional title>, <YYYY>–present (<sub-org if any>)
- **Scope:** $<N>M program portfolio, <N>-person org (<composition>), <stack scope>, <boundary>.
- **Org structure (canonical — do not fabricate variants):** <exact structure>. **Never claim <past anti-pattern>** — past fabrication errors.
- **Title note:** HR title is "<HR title>," actual scope is <level>. <Style-guide phrasing recommendation>.
- **Built / shipped:**
  - <Thing 1 with mechanism and named system>
  - <Thing 2 …>
  - …
- **Outcomes:**
  - <Quantified outcome 1>
  - <Quantified outcome 2>
- **<Optional honesty notes, e.g. agentic systems caveat>**

### Prior roles (compressed for reference)
- **<Company>, <Title>, <date range>.** <1–2 sentences of the most resume-relevant accomplishments.>
- ...

### Strongest single-line credentials
- $<N>M program / <N>-person org (scope)
- <Named production system> (scale + named system)
- ...
```

## Hard rules for this section

- **Pull only what the user states.** Never extrapolate numbers from one figure to another.
- **The "canonical org structure" line is load-bearing.** It's what stops future drafting agents from fabricating variants. Capture it explicitly.
- **Anti-pattern notes belong here.** If the user mentions "a prior resume said I managed 80 engineers but I actually only managed 50" — capture both the truth and the explicit "never claim 80" note.
- **TBD is fine, fabrication is not.** Better an honest gap than an invented number.
