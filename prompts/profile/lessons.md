# Section: Lessons from past interviews

## Canonical heading

`## Lessons from past interviews (use to calibrate)`

## What this section captures

Specific past interviews (or applications) with the **outcome**, **feedback received**, and the **lesson** for future tailoring. The drafting agent reads this to avoid repeating positioning errors.

## Questions to cover

For each notable past interview the user remembers:

1. **Company + role + outcome.** Format like: "&lt;Company A&gt; &lt;Role&gt; — final round, rejected." or "&lt;Company B&gt; &lt;Role&gt; — phone screen, no further contact." or "&lt;Company C&gt; &lt;Role&gt; — 7–8 rounds, rejected on pay/level."
2. **Specific feedback.** What did the interviewer / recruiter / rejection email say? Quote it if you have it.
3. **Lesson for future tailoring.** What's the **implication** — what to change next time? E.g., "Rejection said 'sounds like an EM' — don't shoehorn EM resume into PM apps; build PM-voice resumes deliberately."

Ask about the last 5–10 most relevant attempts, prioritizing ones with **specific feedback** over generic rejections.

If the user has no past interviews to reflect on, this section can be brief or empty for now — and grow over time as the workflow runs.

## Output structure (draft format)

```
## Lessons from past interviews (use to calibrate)

- **<Company> (<Role>) — <rounds count if known>, <outcome>.** <Brief context, feedback if known.> *Implication:* <what to change for future apps>.
- **<Company> (<Role>) — <outcome>.** <Feedback.> *Implication:* <lesson>.
- ...
```

## Hard rules for this section

- **Quote feedback verbatim where possible.** "Sounds like an EM" is more useful than "they thought I wasn't a PM."
- **Every entry has an *Implication* line.** Without that, the agent can't act on the lesson.
- **Keep it factual.** "Rejected on pay/level" not "I think they didn't like me."
